a11ytips.css

This page demonstrates a11ytips.css, a small experiment in flagging accessibility issues using CSS.

This experiment could go into more detail, for example with the various rules involving roles and required attributes, but there are a number of important issues which cannot be highlighted using this method (e.g. heading structure, colour contrast). In short, this approach does not offer enough coverage to be useful.

Below are some of the issues covered, and examples of passing or failing implementations. Issues are flagged with a dotted outline (red for errors, amber for warnings) and a short explanation is shown on hover.

It is important to note that this approach is not itself accessible, as tips are not able to be reached via the keyboard.

Inspired by Heydon Pickering's nifty REVENGE.CSS, which is still a lot of fun.

Controls

Some frequently seen issues with controls include:

  • Buttons or links which contain only background images. These elements are empty to assistive technology. Fix with either on-screen content which is visually hidden, or an accessible label.
  • Tab indexes set to any value other than -1 or 0. These will disrupt the native tab order for keyboard users.
  • Using aria-hidden on elements which are not intended for screen reader users but can still be tabbed to. This will result on users tabbing to "nothing".

Roles

Misuse of roles and associated attributes include:

  • Using roles to create controls where a native element exists. It is always preferable to use native elements wherever possible.
  • Failing to use the required attributes when using a role (e.g. aria-checked with a role of checkbox).
  • Use of the application role. This is almost always a bad idea, and should not be used unless authors have a thorough understanding of the effect.
